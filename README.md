# Пепел и Клятва (Ashes & Oath)

Браузерная стратегия в реальном времени для двух игроков. Игроки создают королевства, развивают экономику, обучают войска и после короткого перемирия сражаются на общей арене. Матч рассчитан примерно на 8–15 минут.

Это запускаемый монорепозиторий, а не макет: React/Vite-клиент подключается к Node.js-серверу через Socket.IO, а все значимые игровые действия проверяет сервер.

## Что входит в игру

- Комнаты по короткому коду, строго два места и запрет третьего подключения.
- Настройка королевства и готовность обоих игроков перед стартом.
- 20-секундное перемирие с закрытой информацией о противнике.
- Серверные ресурсы, здания, уровни ратуши, население и очередь обучения.
- Чистые и смешанные отряды, ограничение числа отрядов, движение и бои.
- Урон зданиям, уничтожение базы, победа/поражение и реванш.
- Переподключение к матчу после краткого разрыва соединения.
- Адаптивный интерфейс, эффекты, звук, управление камерой и отображение сетевого статуса.
- Автоматические тесты ключевых правил и сетевых сценариев.

## Быстрый старт

Понадобятся Node.js 20 или новее и npm 10 или новее. Рекомендуется актуальная LTS-версия Node.js. Команды выполняются из корня репозитория.

~~~powershell
npm install
Copy-Item .env.example .env
npm run dev
~~~

На macOS/Linux вместо Copy-Item:

~~~bash
cp .env.example .env
~~~

После запуска откройте http://localhost:5173. Сервер работает на http://localhost:3001, его проверка состояния доступна по адресу http://localhost:3001/health.

Первый запуск предварительно собирает пакет shared, затем одновременно поднимает клиент и сервер. Остановить оба процесса можно сочетанием Ctrl+C.

## Как сыграть

1. Первый игрок нажимает «Создать комнату» и копирует появившийся код.
2. Второй игрок нажимает «Подключиться», вводит код комнаты и входит.
3. Оба настраивают королевства и подтверждают готовность.
4. Во время перемирия развивайте добычу, жильё и армию. Противник и его владения скрыты.
5. После окончания таймера формируйте отряды, выбирайте цель на арене и отправляйте войска.
6. Побеждает игрок, разрушивший главную базу противника, либо оставшийся в матче после истечения окна переподключения соперника.

Клиент лишь отправляет намерение игрока и показывает подтверждённое состояние. Изменить ресурсы, закончить перемирие раньше времени или нанести произвольный урон через консоль браузера нельзя: сервер повторно проверяет действие.

## Архитектура

~~~text
ashes-and-oath/
├── client/                 React + TypeScript + Vite, Phaser, UI и сетевой клиент
├── server/                 Node.js + Socket.IO, комнаты и авторитетная симуляция
├── shared/                 общие типы, протокол событий и настройки баланса
├── assets/                 исходные и сгенерированные графические материалы
├── Dockerfile.client       сборка SPA и раздача через nginx
├── Dockerfile.server       сборка и запуск Node.js-сервера
├── nginx.conf              SPA fallback и reverse proxy для Socket.IO
├── docker-compose.yml      production-подобный запуск двух контейнеров
├── tsconfig.base.json      базовые строгие настройки TypeScript для workspace-конфигов
└── package.json            npm workspaces и корневые команды
~~~

    end
~~~

### Ответственность слоёв

**client** отвечает за меню, лобби, отрисовку арены, ввод, анимации, звуки, интерполяцию и индикацию соединения. Клиентское предсказание, если используется, носит только визуальный характер.

**server** хранит комнаты в памяти, ведёт игровые часы, начисляет ресурсы, завершает строительство и обучение, перемещает отряды, считает бой, фильтрует туман войны и определяет победителя. Клиентское время и присланные числовые результаты не считаются доверенными.

**shared** задаёт единый контракт событий, снимков состояния и баланса, поэтому клиент и сервер компилируются против одних типов.

> Текущее хранилище комнат — оперативная память одного серверного процесса. Перезапуск удаляет активные матчи. Для горизонтального масштабирования недостаточно просто включить несколько реплик: потребуется общее хранилище состояния/координатор симуляции и Socket.IO adapter. До этого сервер следует запускать в одной реплике.

## Проверка перед публикацией

~~~powershell
npm ci
npm run typecheck
npm test
npm run build
~~~

После сборки:

- client/dist содержит статический SPA;
- shared/dist содержит общий исполняемый код и декларации;
- server/dist содержит production-сервер;
- npm start запускает только Node.js-сервер. Статический клиент в обычной production-схеме раздаёт nginx, CDN или static-site хостинг.

Для локальной проверки полного production-подобного варианта проще использовать Docker Compose, описанный ниже.

Native production-запуск сервера без Docker в PowerShell:

~~~powershell
npm ci
$env:VITE_SERVER_URL = "https://api.game.example.com"
npm run typecheck
npm test
npm run build
$env:NODE_ENV = "production"
$env:HOST = "0.0.0.0"
$env:PORT = "3001"
$env:CLIENT_ORIGIN = "https://game.example.com"
npm start
~~~

На macOS/Linux:

~~~bash
npm ci
export VITE_SERVER_URL=https://api.game.example.com
npm run typecheck && npm test && npm run build
export NODE_ENV=production HOST=0.0.0.0 PORT=3001
export CLIENT_ORIGIN=https://game.example.com
npm start
~~~

Раздавайте client/dist отдельным nginx/CDN и направляйте VITE_SERVER_URL на публичный HTTPS-адрес сервера либо настройте same-origin proxy. Команда vite preview предназначена для локальной проверки сборки, а не как постоянный интернет-сервер.

## Ручное тестирование мультиплеера

### Две вкладки

1. Запустите npm run dev.
2. Откройте http://localhost:5173 в первой вкладке и создайте комнату.
3. Откройте этот же адрес во второй, отдельно созданной вкладке, и войдите по коду.
4. Не дублируйте уже открытую игровую вкладку: некоторые браузеры копируют session storage при дублировании.
5. Проверьте готовность, старт перемирия, синхронное окончание таймера, строительство, обучение, создание отряда и бой.
6. Обновите одну вкладку во время матча и убедитесь, что переподключение восстанавливает её место.
7. Попробуйте открыть третью вкладку и войти по тому же коду: сервер должен отклонить вход.

### Два разных браузера

Откройте клиент, например, в Chrome и Firefox/Edge. Этот вариант лучше имитирует двух независимых пользователей: хранилища браузеров, кэш и Socket.IO-соединения не пересекаются.

### Два компьютера в одной локальной сети

Узнайте IPv4-адрес компьютера, где запущен проект:

~~~powershell
ipconfig
~~~

Предположим, адрес — 192.168.1.42. Перед запуском задайте публичные для локальной сети адреса:

~~~powershell
$env:VITE_SERVER_URL = "http://192.168.1.42:3001"
$env:CLIENT_ORIGIN = "http://192.168.1.42:5173"
npm run dev
~~~

На обоих компьютерах открывайте именно http://192.168.1.42:5173, включая компьютер-хост. Разрешите входящие TCP-подключения к портам 5173 и 3001 в Windows Defender Firewall. Устройства должны быть в одной сети без включённой изоляции клиентов Wi-Fi.

Более простой LAN-вариант через Docker использует один публичный порт:

~~~powershell
Copy-Item .env.example .env
# В .env: PUBLIC_ORIGIN=http://192.168.1.42:8080
docker compose up --build -d
~~~

После этого оба игрока открывают http://192.168.1.42:8080. Socket.IO проходит через тот же адрес и порт.

## Переменные окружения

| Переменная | По умолчанию | Где используется | Описание |
| --- | --- | --- | --- |
| NODE_ENV | development | сервер | development или production |
| HOST | 0.0.0.0 | сервер | Интерфейс прослушивания; 0.0.0.0 нужен контейнеру и LAN |
| PORT | 3001 | сервер | Внутренний HTTP/Socket.IO-порт |
| CLIENT_ORIGIN | http://localhost:5173 | сервер | Точный origin, разрешённый CORS |
| VITE_SERVER_URL | http://localhost:3001 | сборка клиента | Публичный адрес Socket.IO; пустое значение означает origin страницы |
| PUBLIC_ORIGIN | http://localhost:8080 | Docker Compose | Origin nginx-клиента, передаваемый серверу как CLIENT_ORIGIN |
| CLIENT_BIND | 0.0.0.0 | Docker Compose | Интерфейс публикации nginx; для LAN нужен 0.0.0.0 |
| CLIENT_PORT | 8080 | Docker Compose | Порт nginx на машине-хосте |
| SERVER_BIND | 127.0.0.1 | Docker Compose | Интерфейс диагностического порта сервера; по умолчанию только localhost |
| SERVER_PORT | 3001 | Docker Compose | Диагностический порт сервера на машине-хосте |

VITE_SERVER_URL — build-time переменная Vite. Изменение значения после npm run build не переписывает готовые файлы: клиент нужно собрать заново. В Docker Compose она намеренно пуста, поэтому браузер подключается к своему origin, а nginx проксирует /socket.io/ во внутренний контейнер server.

Для временных значений в PowerShell:

~~~powershell
$env:PORT = "3001"
$env:CLIENT_ORIGIN = "http://localhost:5173"
$env:VITE_SERVER_URL = "http://localhost:3001"
npm run dev
~~~

Корневой .env читают Docker Compose, сервер и Vite (через envDir). Значения из shell имеют приоритет и удобны для разовых LAN-проверок. После изменения VITE_SERVER_URL перезапустите Vite, потому что клиент получает эту переменную при запуске/сборке. Не коммитьте .env: он исключён через .gitignore.

## CORS, Socket.IO и HTTPS

CLIENT_ORIGIN должен полностью совпадать с адресом в строке браузера: важны схема, домен и порт. Например, https://game.example.com и http://game.example.com — разные origin. Несколько точных origin можно перечислить через запятую. В production не используйте wildcard вместо известных доменов.

Socket.IO работает на стандартном пути /socket.io/, предпочитает WebSocket и допускает HTTP long-polling как резервный транспорт. Reverse proxy обязан пропускать и обычные запросы, и заголовки Upgrade/Connection. Корневой nginx.conf уже настроен для обоих транспортов и увеличенного тайм-аута соединения.

При HTTPS страницы адрес сервера также должен быть HTTPS; браузер заблокирует небезопасное HTTP/WebSocket-соединение как mixed content. При same-origin схеме достаточно открыть сайт по HTTPS: Socket.IO автоматически использует защищённый WSS через тот же домен.

Если позже появятся несколько серверных реплик, учтите две отдельные задачи:

1. Long-polling требует привязки клиента к одной реплике либо отказа от polling.
2. Комнаты и симуляция должны быть вынесены из памяти процесса в согласованное общее хранилище/координатор.

Один только load balancer или Redis adapter не превращает локальное состояние игры в распределённое.

## Health checks

Сервер предоставляет GET /health. Успешный ответ 2xx означает, что HTTP-процесс отвечает; JSON содержит status, service, uptime, число комнат/игроков и timestamp. nginx также проксирует /health на сервер, поэтому внешний монитор проверяет сразу публичный путь.

~~~powershell
curl.exe -f http://localhost:3001/health
curl.exe -f http://localhost:8080/health
~~~

Docker:

~~~powershell
docker compose ps
docker inspect --format "{{json .State.Health}}" $(docker compose ps -q server)
~~~

Имя контейнера зависит от имени папки/проекта, поэтому обычно удобнее смотреть docker compose ps. Для внешнего мониторинга укажите https://ваш-домен/health с интервалом 30–60 секунд. Health check не гарантирует, что конкретный матч исправен; для production-наблюдаемости дополнительно нужны метрики ошибок, числа соединений и задержки игрового цикла.

## Docker Compose

Понадобятся Docker Desktop либо Docker Engine с Compose v2.

~~~powershell
Copy-Item .env.example .env
docker compose up --build -d
docker compose ps
docker compose logs -f --tail=200
~~~

Игра будет доступна на http://localhost:8080. Схема контейнеров:

~~~text
браузер
   │ http://localhost:8080
   ▼
client (nginx: SPA + /socket.io/ + /health)
   │ внутренняя Docker-сеть
   ▼
server (Node.js :3001)
~~~

Полезные команды:

~~~powershell
# Перезапустить только сервер
docker compose restart server

# Пересобрать и применить изменения
docker compose up --build -d

# Остановить, сохранив образы
docker compose down

# Посмотреть последние ошибки
docker compose logs --tail=200 server
docker compose logs --tail=200 client
~~~

Остановка или перезапуск server завершает активные матчи, потому что комнаты находятся в памяти. Планируйте обновления в период без игроков.

## Публикация игры

### Варианты

Короткая рекомендация: для первого облачного запуска выберите Render; для бесплатной/условно бесплатной проверки можно временно использовать его Static Site и Free Web Service, принимая засыпание и лимиты; для стабильных матчей нужен always-on платный Web Service либо небольшой VPS с Docker Compose.

| Вариант | Когда подходит | Важные оговорки |
| --- | --- | --- |
| Railway | Быстрый Git/Docker-деплой сервера, удобные переменные и домены | В актуальной документации указан лимит длительности одного WebSocket-запроса; Socket.IO переподключится, но сценарий нужно обязательно проверить на полном матче |
| Render | Удобная связка Static Site для client и Web Service для server | Бесплатные сервисы могут засыпать после периода без входящего HTTP/WebSocket-трафика; пробуждение или замена instance прерывает матч |
| Fly.io | Контейнеры по регионам, гибкая сеть и CLI | Потребуется настроить Machines и autostop; оставьте минимум одну работающую Machine и не включайте несколько независимых игровых реплик |
| VPS + Docker Compose | Предсказуемая одна машина, полный контроль и переносимость | Вы отвечаете за Linux, firewall, TLS, обновления, мониторинг и резервные копии |
| Обычный VPS без Docker | Минимальный overhead, запуск Node.js через systemd/PM2 | Больше ручной настройки и выше риск расхождения окружений |
| Клиент отдельно на CDN/static hosting | Самая дешёвая и быстрая раздача SPA | VITE_SERVER_URL встраивается при сборке; CORS сервера должен разрешать домен клиента |

Тарифы и ограничения хостингов меняются. Перед выбором проверьте в актуальной документации поддержку долгоживущих WebSocket, отсутствие обязательного засыпания, лимиты памяти/CPU/исходящего трафика, пользовательские домены и правила health check. Для этой версии критично держать ровно один серверный экземпляр. На момент подготовки README (июль 2026 года) Railway документирует 15-минутный предел одного WebSocket-соединения; Socket.IO умеет переподключаться, а игра сохраняет место игрока ограниченное время, но этот переход следует нагрузочно проверить до публикации.

### Рекомендуемый путь для новичка: Render Web Service + Static Site

На момент подготовки README (июль 2026 года) Render — наиболее прямой облачный путь для новичка: Web Service принимает входящие WebSocket, слушает назначенный PORT, поддерживает /health, custom domains и управляемый TLS, а Static Site раздаёт готовый client/dist. Названия пунктов панели могут меняться, поэтому сверяйтесь с официальными ссылками выше. Та же двухсервисная схема применима к аналогичным платформам.

1. Создайте Git-репозиторий и отправьте туда проект вместе с package-lock.json, но без .env.
2. Зарегистрируйтесь на выбранной платформе и подключите Git-провайдера.
3. Создайте server Web Service из этого репозитория.
4. Для Docker-варианта укажите Dockerfile.server. Для native Node-варианта используйте:

   ~~~text
   Build: npm ci && npm run build -w @ashes/shared --if-present && npm run build -w @ashes/server
   Start: npm run start -w @ashes/server
   Health: /health
   ~~~

5. Задайте NODE_ENV=production, HOST=0.0.0.0 и CLIENT_ORIGIN, временно оставив в CLIENT_ORIGIN будущий адрес клиентского сайта. PORT обычно назначает сама платформа; не переопределяйте системный PORT, если документация сервиса этого не требует.
6. Оставьте одну server-реплику, отключите scale-to-zero/autosleep для реальных матчей и дождитесь публичного HTTPS-адреса, например https://ashes-api.example-host.app.
7. Создайте Static Site из того же репозитория:

   ~~~text
   Build: npm ci && npm run build -w @ashes/shared --if-present && npm run build -w @ashes/client
   Publish directory: client/dist
   Build env: VITE_SERVER_URL=https://ashes-api.example-host.app
   ~~~

8. Получите HTTPS-адрес клиента и установите его точным значением CLIENT_ORIGIN на сервере, например https://ashes.example-host.app. Перезапустите server.
9. Откройте два браузера, создайте комнату, проверьте вход второго игрока, /health и переподключение.
10. Подключите собственные домены в панели платформы. После смены адреса сервера обновите VITE_SERVER_URL и пересоберите клиент; после смены адреса клиента обновите CLIENT_ORIGIN.

На managed-платформе не используйте Dockerfile.client, если клиент и сервер не находятся в одной приватной сети с DNS-именем server. Для раздельного размещения проще и надёжнее публиковать client/dist как Static Site и встраивать публичный VITE_SERVER_URL.

Socket.IO автоматически пытается восстановить транспорт после сетевого разрыва, а игра резервирует место игрока на короткое окно. Но замена/redeploy самого Render instance уничтожает все комнаты в его памяти, поэтому текущий матч восстановить после такого события нельзя. Публикуйте новую версию в период без активных игроков и оставляйте одну server-replica.

### VPS + Docker Compose

Для постоянной небольшой игры это часто самый предсказуемый вариант.

1. Создайте VPS с актуальным Ubuntu/Debian, добавьте SSH-ключ и обычного пользователя с sudo.
2. В DNS создайте A-запись game.example.com на IPv4 VPS; AAAA добавляйте только при реально настроенном IPv6.
3. Установите Docker Engine и Compose plugin по официальной инструкции вашего дистрибутива.
4. Разрешите firewall только SSH, HTTP и HTTPS. Порт 3001 не нужно открывать в интернет.
5. Клонируйте репозиторий, создайте .env и задайте:

   ~~~dotenv
   NODE_ENV=production
   PUBLIC_ORIGIN=https://game.example.com
   CLIENT_BIND=127.0.0.1
   CLIENT_PORT=8080
   SERVER_BIND=127.0.0.1
   SERVER_PORT=3001
   ~~~

6. Запустите docker compose up --build -d и проверьте curl -f http://127.0.0.1:8080/health.
7. Поставьте перед контейнером Caddy, nginx или Traefik. Минимальный Caddyfile:

   ~~~caddyfile
   game.example.com {
     reverse_proxy 127.0.0.1:8080
   }
   ~~~

8. Caddy автоматически получает и обновляет сертификат при корректном DNS и открытых портах 80/443. Для nginx настройте сертификат через ACME-клиент и обязательно передайте WebSocket Upgrade.
9. Проверьте https://game.example.com/health и сыграйте тестовый матч в двух браузерах.

Для дополнительной изоляции привяжите CLIENT_PORT только к 127.0.0.1 в compose или firewall, если внешний reverse proxy работает на той же машине. Сам Node-порт также следует оставлять доступным только локально/в Docker-сети.

## Домен и HTTPS

На managed-платформе добавьте домен в настройках сервиса, затем создайте предложенную CNAME/A-запись. Сертификат обычно выпускается автоматически, но точный процесс зависит от провайдера.

На VPS:

- направьте DNS на сервер;
- откройте 80/443;
- используйте Caddy/Traefik или nginx + ACME;
- задайте PUBLIC_ORIGIN=https://домен для Docker Compose;
- не публикуйте 3001 наружу без необходимости;
- проверьте сертификат, /health и WebSocket через инструменты разработчика браузера.

Изменение DNS может распространяться не мгновенно. Не включайте HTTPS-редирект до того, как сертификат успешно получен, если выбранный reverse proxy не управляет этим автоматически.

## Логи, перезапуск и обновление

Локально сервер пишет логи в терминал. В Docker:

~~~powershell
docker compose logs -f --tail=200 server
docker compose logs --since=30m server
docker compose restart server
~~~

На managed-платформе используйте раздел Logs/Events и настройте оповещение о неуспешном /health. На VPS следите также за диском, RAM, load average и журналами reverse proxy.

Безопасное обновление Docker-установки:

~~~bash
cd /opt/ashes-and-oath
git pull --ff-only
docker compose build --pull
docker compose up -d
docker compose ps
curl -f https://game.example.com/health
~~~

Перед обновлением предупредите игроков: сервер не сохраняет текущие комнаты между перезапусками. Для отката переключитесь на предыдущий проверенный commit/tag и снова выполните docker compose up --build -d.

Параметр restart: unless-stopped в docker-compose.yml автоматически поднимает контейнеры после сбоя и перезагрузки Docker-хоста. Он не исправляет бесконечный crash loop — причину нужно искать в docker compose logs.

## Резервное копирование

В текущей версии нет базы данных и постоянных профилей, поэтому игровые комнаты резервировать невозможно и не требуется: они намеренно временные. Сохраняйте:

- Git-репозиторий и проверенные release tags;
- зашифрованную копию production .env вне репозитория;
- конфигурацию DNS, firewall и reverse proxy;
- инструкции развертывания и сведения о выбранном тарифе;
- при появлении БД — автоматические snapshots/dumps и отдельную проверку восстановления.

Никогда не считайте snapshot единственной копией. Резервные копии должны храниться отдельно от VPS, иметь ограниченный доступ и периодически проходить тестовое восстановление.

## Диагностика

**Клиент показывает «нет соединения».** Проверьте http://localhost:3001/health, значение VITE_SERVER_URL и вкладку Network → WS в DevTools. После изменения VITE_SERVER_URL перезапустите Vite или пересоберите клиент.

**Ошибка CORS.** Скопируйте origin из адресной строки целиком и задайте его как CLIENT_ORIGIN без пути. Проверьте схему и порт. После изменения перезапустите server.

**На втором компьютере клиент открывается, но сервер недоступен.** Нельзя оставлять localhost в VITE_SERVER_URL: для удалённого браузера localhost означает сам второй компьютер. Используйте LAN-IP хоста либо Docker same-origin схему.

**WebSocket получает 400/404/502.** Убедитесь, что proxy передаёт /socket.io/, HTTP/1.1, Upgrade и Connection, а тайм-аут больше обычного периода ping. Проверьте логи server и reverse proxy.

**Docker-контейнер unhealthy.**

~~~powershell
docker compose ps
docker compose logs --tail=200 server
docker compose logs --tail=200 client
curl.exe -v http://localhost:8080/health
~~~

**Порт занят.** Остановите конфликтующий процесс либо измените PORT/VITE_SERVER_URL для dev или CLIENT_PORT/SERVER_PORT для Compose.

**После обновления виден старый интерфейс.** Убедитесь, что пересобран client, затем выполните hard reload. index.html отдаётся без кэширования; /assets имеет короткий кэш, поскольку там есть как хэшированные bundles, так и файлы public с постоянными именами.
